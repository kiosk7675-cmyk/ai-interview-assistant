# 测试开发面经

## 测试理论

### 测试分类
- 按阶段：单元测试（函数级）→ 集成测试（模块间）→ 系统测试（全链路）→ 验收测试（用户级）
- 按方式：黑盒（功能测试，不关心内部实现）、白盒（代码级路径覆盖）、灰盒（部分了解内部）
- 按目的：功能测试、性能测试、安全测试、兼容性测试、回归测试

### 测试用例设计
等价类划分：有效/无效等价类。边界值分析：边界及边界附近。判定表：多条件组合。状态迁移图：状态变化路径。正交实验法：减少组合数。错误推测法：基于经验。

### 软件测试原则
1. 测试证明缺陷存在，而非证明无缺陷
2. 尽早测试（Shift-Left）
3. 杀虫剂悖论：反复用同样用例会降低发现缺陷能力
4. 测试依赖于上下文
5. 缺陷集群性（帕累托法则：80% 缺陷集中在 20% 模块）
6. 不要穷尽测试

## 自动化测试

### 单元测试
Python：pytest（fixture/parametrize/conftest.py）、unittest（标准库）。Java：JUnit 5（@Test/@BeforeEach/@ParameterizedTest）、Mockito（mock 对象）。覆盖率：语句覆盖、分支覆盖、路径覆盖。目标：核心逻辑 ≥80%，非核心 ≥60%。

### 接口自动化
Python：requests + pytest，allure 报告。Java：RestAssured。关键设计：数据驱动（CSV/YAML/数据库）、环境切换（配置文件）、鉴权处理（Token 刷新）、数据清理（前后置处理）、断言策略（状态码+业务码+字段校验）。

### UI 自动化
Selenium：WebDriver 协议，支持多浏览器。Playwright：更现代，自动等待，支持多标签页/网络拦截。Appium：移动端 UI 自动化。PO 模式（Page Object）：将页面元素和操作封装为类，提高可维护性。关键挑战：元素定位不稳定、动态加载、跨浏览器兼容。

### Mock 与 Stub
Mock：模拟外部依赖行为，验证交互。Stub：返回预设数据，不验证交互。Python：`unittest.mock`（patch/Mock/MagicMock）。Java：Mockito/WireMock。前端：MSW（Mock Service Worker）。

## 性能测试

### 性能测试类型
负载测试：逐步加压到预期指标。压力测试：超过预期负载，找系统极限。稳定性测试：长时间运行，发现内存泄漏。基准测试：单接口/单模块性能基线。

### JMeter 核心概念
线程组：模拟并发用户。取样器：HTTP 请求/JDBC/JSR223。断言：响应码/内容/JSON Path。监听器：聚合报告/图形结果。参数化：CSV Data Set Config。分布式压测：多机协同产生高并发。

### 性能指标
TPS/QPS：每秒事务/查询数。响应时间：平均/P90/P99。并发数：同时在线用户数。吞吐量：单位时间处理请求量。错误率：失败请求占比。资源利用率：CPU/内存/IO/网络。

### 性能调优思路
定位瓶颈：CPU 密集（代码优化/算法优化）、IO 密集（缓存/批量/异步）、内存（泄漏/GC 频繁）。工具：top/htop、iostat、vmstat、Arthas（Java）、py-spy（Python）、pprof（Go）。数据库慢查询：EXPLAIN 分析执行计划、索引优化。

## 持续集成测试

### 测试金字塔
单元测试（70%）：快速、稳定、低成本。接口测试（20%）：验证模块间交互。UI 测试（10%）：慢、脆弱、高维护成本。原则：尽量用低成本测试覆盖更多场景。

### 测试左移
在需求/设计阶段就介入测试。代码审查中的测试思维：边界条件、异常处理、并发安全。TDD（测试驱动开发）：先写测试→实现代码→重构。BDD（行为驱动开发）：Given-When-Then 自然语言描述行为。

### 质量度量
缺陷密度（缺陷数/KLOC）、缺陷发现率（测试发现/总缺陷）、用例通过率、自动化覆盖率、线上故障率。质量门禁：合并前必须通过全部自动化测试+代码覆盖率达标+静态扫描无高危。
