# 系统设计与项目实战

## 什么是 RAG？如何实现？
RAG（Retrieval-Augmented Generation）= 检索增强生成，将外部知识检索与 LLM 生成结合，减少幻觉。
- **流程**：用户提问 → 向量检索相关文档 → 拼接 prompt（问题+检索结果）→ LLM 生成回答。
- **向量嵌入**：将文本转为高维向量，相似语义的文本距离近。模型如 BGE、text-embedding-3。
- **向量数据库**：存储和检索向量，如 ChromaDB、Milvus、Qdrant、FAISS。
- **文档切分**：长文档按段落/固定长度切分，每片 200-500 字，overlap 50 字防止信息断裂。
- **优化**：混合检索（向量+关键词 BM25）、重排序（Reranker）、查询改写。

## 什么是 Agent？ReAct 模式？
- **Agent**：能自主决策、调用工具、规划步骤的 AI 系统。核心是"感知→推理→行动"循环。
- **ReAct**：Reasoning + Acting 交替进行。模型先思考（Thought），再选择工具执行（Action），观察结果（Observation），循环直到得到最终答案。
- **与 RAG 的区别**：RAG 是单次检索+生成，Agent 可以多步推理、多轮检索、串行调用多个工具。
- **LangChain/LangGraph**：构建 Agent 的框架，LangGraph 支持状态图、循环、条件分支。

## MCP（Model Context Protocol）是什么？
- **定义**：Anthropic 提出的开放协议，标准化 LLM 与外部工具/数据源的交互方式。
- **类比**：类似 USB-C 协议，统一了 AI 应用连接各种工具的接口。
- **核心概念**：Tools（可调用的函数）、Resources（可读取的数据）、Prompts（预定义提示词）。
- **实现**：Python SDK `mcp` 包，定义 `@server.tool` 装饰器暴露工具，客户端通过 JSON-RPC 调用。

## 短链接系统设计？
- **核心流程**：长 URL → 哈希/自增ID → 短码 → 存储（短码→长URL映射）→ 302 重定向。
- **短码生成**：自增 ID + Base62 编码（0-9a-zA-Z），6位可表示 568 亿。
- **存储**：MySQL（简单）或 Redis（快）+ 布隆过滤器防重复。
- **高并发**：缓存热点短码、读写分离、分库分表。
- **防滥用**：频率限制、黑名单、长 URL 合法性校验。

## 限流算法？
- **固定窗口**：1 分钟内最多 N 次，窗口边界有突发问题（两倍流量）。
- **滑动窗口**：将窗口细分为小格，精确控制，实现稍复杂。
- **令牌桶**：固定速率放入令牌，请求消耗令牌，桶满丢弃。允许突发（桶内有令牌时）。
- **漏桶**：固定速率处理请求，超出排队或丢弃。平滑输出，不允许突发。
- **Redis 实现**：INCR + EXPIRE（固定窗口）、ZSet（滑动窗口）、令牌桶用 Lua 脚本。

## 消息队列的应用场景？
- **异步处理**：注册后异步发邮件/短信，提高响应速度。
- **流量削峰**：秒杀场景，请求先入队，消费者按能力处理。
- **解耦**：订单系统发消息，库存/物流/积分各系统独立消费。
- **常见 MQ**：RabbitMQ（AMQP，功能丰富）、Kafka（高吞吐，日志场景）、Redis Stream（轻量级）。

## 如何设计一个安全的 API？
- **认证**：JWT / OAuth2，Token 存 HttpOnly Cookie 或 Authorization Header。
- **授权**：RBAC（基于角色）或 ABAC（基于属性），最小权限原则。
- **输入验证**：Pydantic 严格校验，拒绝非法输入。
- **SQL 注入**：ORM 参数化查询，禁止拼接 SQL。
- **XSS**：响应头 `Content-Type: application/json`，前端不用 v-html 渲染用户内容或使用消毒库。
- **CSRF**：SameSite Cookie、CSRF Token。
- **限流**：防止暴力破解和 DDoS。
- **HTTPS**：全站 HTTPS，HSTS。
- **日志**：记录操作审计日志，不记录敏感信息（密码、Token）。

## 项目简历怎么写？（STAR 法则）
- **Situation**：项目背景，解决什么问题。
- **Task**：你的职责，负责什么模块。
- **Action**：用了什么技术，怎么实现的，遇到了什么问题怎么解决。
- **Result**：量化成果（QPS 提升 XX%、响应时间降低 XX%、覆盖 XX 接口）。
- **切忌**：只列技术栈，没有细节；夸大不存在的功能；写团队成果当个人贡献。

## 面试常见行为题？
- **项目冲突**：描述技术分歧 → 沟通过程 → 数据/实验驱动决策 → 最终结果。
- **最大挑战**：描述困难 → 分析原因 → 解决方案 → 学到了什么。
- **学习方式**：具体说最近学了什么、怎么学的（文档/源码/项目）、应用到什么场景。
- **团队协作**：强调沟通方式、代码规范、Code Review、文档先行。
