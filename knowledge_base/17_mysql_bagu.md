# MySQL 数据库八股文

## InnoDB 和 MyISAM 的区别？
- **InnoDB**：支持事务、行级锁、外键、MVCC、崩溃恢复（redo log）。默认引擎，推荐使用。
- **MyISAM**：不支持事务和行锁，表级锁，读取快，支持全文索引（5.6前）。适合读多写少场景。
- **关键区别**：InnoDB 聚簇索引存数据，MyISAM 索引和数据分开存储；InnoDB 支持崩溃安全恢复，MyISAM 需 repair 修复。

## B+ 树为什么适合数据库索引？
- **对比 B 树**：B+ 树非叶子节点只存键值（不存数据），单个节点能存更多键，树更矮，磁盘 I/O 更少。
- **对比 Hash**：Hash 不支持范围查询和排序，B+ 树叶子节点链表连接，范围查询高效。
- **对比红黑树**：红黑树太高（O(logN) 但常数大），磁盘 I/O 次数多；B+ 树矮胖（通常3-4层存千万数据）。
- **叶子链表**：便于范围查询和 ORDER BY。

## 为什么推荐自增主键？
- **页分裂**：非自增主键插入时可能触发页分裂（B+树节点重排），性能下降。
- **自增主键**：顺序写入，追加到页末尾，无页分裂，写入性能最优。
- **UUID 问题**：36字符占空间大、随机插入导致页分裂、无序无法利用聚簇索引局部性。

## 什么是回表？如何避免？
- **回表**：通过二级索引查到主键值，再回聚簇索引查完整行数据，两次 B+ 树查找。
- **避免方式**：覆盖索引——查询的列全在索引中，`EXPLAIN` 显示 `Using index`。
- **示例**：`SELECT name FROM user WHERE age=20`，在 `(age, name)` 联合索引上可直接返回，无需回表。

## 最左前缀原则？
- **联合索引 (a, b, c)**：可匹配 `a`、`a,b`、`a,b,c` 查询，不能跳过 a 直接查 `b,c`。
- **原因**：B+ 树按定义顺序排列，先按 a 排序，a 相同再按 b 排序，以此类推。
- **范围查询**：`WHERE a=1 AND b>2 AND c=3`，c 无法利用索引（b 范围查询后的列失效）。
- **优化**：将等值查询列放前面，范围查询列放后面。

## 索引失效的常见场景？
- 对索引列使用函数：`WHERE YEAR(create_time) = 2024` → 改为 `WHERE create_time >= '2024-01-01'`
- 隐式类型转换：`WHERE varchar_col = 123` → MySQL 会对列做类型转换导致失效
- LIKE 左模糊：`WHERE name LIKE '%张'` → 全表扫描
- OR 条件：OR 一侧无索引则全表扫描
- NOT IN / NOT EXISTS / != / <> 通常不走索引
- 联合索引不满足最左前缀

## 事务的隔离级别与并发问题？
| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| READ UNCOMMITTED | ✓ | ✓ | ✓ |
| READ COMMITTED | ✗ | ✓ | ✓ |
| REPEATABLE READ | ✗ | ✗ | ✓（InnoDB 通过 MVCC+Next-Key Lock 基本解决） |
| SERIALIZABLE | ✗ | ✗ | ✗ |

## MVCC 实现原理？
- **隐藏列**：`DB_TRX_ID`（最后修改的事务ID）、`DB_ROLL_PTR`（回滚指针指向 undo log）
- **Read View**：记录创建时活跃事务列表，用于判断某版本是否对当前事务可见
- **可见性判断**：如果版本的 trx_id < Read View 最小活跃事务ID → 可见；如果在活跃列表中 → 不可见，沿 undo log 链找更早版本
- **RC vs RR**：RC 每次 SELECT 创建新 Read View；RR 只在首次 SELECT 创建

## MySQL 日志系统：redo log、undo log、binlog？
- **redo log**：InnoDB 引擎层，物理日志，记录"某页某偏移量修改为某值"，崩溃恢复用，WAL 机制先写日志再写磁盘
- **undo log**：InnoDB 引擎层，逻辑日志，记录数据修改前的值，用于事务回滚和 MVCC
- **binlog**：Server 层，逻辑日志，记录所有 DDL/DML 操作，用于主从复制和数据恢复
- **两阶段提交**：redo log prepare → binlog 写入 → redo log commit，保证主从一致性

## 慢查询优化步骤？
1. 开启慢查询日志：`slow_query_log=1, long_query_time=1`
2. 用 `EXPLAIN` 分析执行计划
3. 关注：type（ALL=全扫描需优化）、key（是否命中索引）、rows（扫描行数）、Extra（Using filesort/Using temporary 需优化）
4. 优化手段：加索引、覆盖索引、避免索引失效写法、分页优化、分表分库

## 分库分表策略？
- **垂直拆分**：按业务模块拆库（用户库、订单库），降低单库压力
- **水平拆分**：按规则（如 user_id % 8）拆表，均匀分布数据
- **分片键选择**：高频查询字段、分布均匀、避免跨片查询
- **问题**：分布式事务、跨片 JOIN、全局 ID（雪花算法）、数据迁移

## MySQL 主从复制原理？
- **异步复制**：主库提交事务后写 binlog，从库 IO 线程拉取 binlog 写 relay log，SQL 线程回放
- **半同步复制**：主库等待至少一个从库确认收到 binlog 才返回成功
- **GTID 复制**：每个事务有全局唯一 ID，简化主从切换和故障恢复
- **延迟原因**：从库单线程回放、大事务、网络延迟、从库性能差
