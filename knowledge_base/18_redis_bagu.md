# Redis 缓存八股文

## Redis 为什么用单线程还这么快？
- **纯内存**：数据在内存中，读写延迟在纳秒级
- **I/O 多路复用**：epoll 机制，单线程监听大量连接，事件驱动
- **避免锁开销**：单线程无锁竞争、无线程切换、无死锁
- **高效数据结构**：SDS 预分配+惰性释放、ziplist 小数据紧凑、skiplist O(logN)
- **注意**：Redis 6.0 引入多线程 I/O（网络读写），命令执行仍是单线程

## Redis String 的 SDS 和 C 字符串区别？
- **O(1) 获取长度**：SDS 有 len 字段，C 字符串需遍历 O(N)
- **二进制安全**：SDS 用 len 判断结束，不依赖 \0，可存任意数据
- **空间预分配**：修改时多分配空间（<1MB 翻倍，≥1MB 加1MB），减少重分配
- **惰性释放**：缩短字符串不立即释放内存，供后续使用

## Redis ZSet 底层实现？
- **元素少时**：ziplist（压缩列表），紧凑存储省内存
- **元素多时**：skiplist + dict，skiplist 支持范围查询 O(logN)，dict 支持单元素 O(1) 查找
- **跳表原理**：多层索引，类似二分查找，空间换时间。平均 O(logN)，最坏 O(N)
- **为什么不用红黑树**：跳表实现简单、范围查询方便（链表遍历）、插入删除只需修改相邻节点

## Redis 过期删除策略？
- **惰性删除**：访问 key 时检查是否过期，过期则删除。CPU 友好但内存不友好
- **定期删除**：每 100ms 随机抽取部分 key 检查过期，过期则删除。折中方案
- **Redis 采用**：惰性删除 + 定期删除
- **注意**：仍有大量过期 key 未被访问也未被定期抽到 → 占用内存 → 需要内存淘汰策略

## Redis 内存淘汰策略？
- **noeviction**：默认，内存满拒绝写入（报错）
- **allkeys-lru**：从所有 key 中淘汰最近最少使用的（推荐）
- **allkeys-lfu**：从所有 key 中淘汰最不经常使用的（Redis 4.0+）
- **volatile-lru**：从设置了过期时间的 key 中淘汰 LRU
- **volatile-lfu**：从设置了过期时间的 key 中淘汰 LFU
- **volatile-ttl**：从设置了过期时间的 key 中淘汰 TTL 最短的
- **allkeys-random**：随机淘汰

## Redis 集群方案？
- **主从复制**：一主多从，读写分离，主写从读。主挂了需手动切换
- **哨兵模式**：Sentinel 监控主节点，自动故障转移（选举新主）
- **Redis Cluster**：官方分布式方案，16384 个 hash slot 分配到多个节点，支持自动分片和故障转移
- **Codis**：豌豆荚开源，代理层分片，对客户端透明，已不维护

## Redis 分布式锁怎么实现？
- **基本实现**：`SET key value NX EX 30`（原子操作，不存在时设置+过期时间）
- **value 用 UUID**：释放时验证 value 确保是自己的锁，防止误删别人的锁
- **Lua 释放锁**：`if redis.call('get',KEYS[1]) == ARGV[1] then return redis.call('del',KEYS[1]) else return 0 end`
- **Redlock**：多节点加锁，半数以上成功才算获取锁，解决单点故障
- **问题**：锁续期（看门狗机制）、可重入、集群模式主从切换丢锁

## Redis 和数据库双写一致性？
- **Cache Aside（旁路缓存）**：读时先读缓存，未命中读数据库并回填；写时先更新数据库再删缓存
- **先更新数据库再删缓存**：删缓存失败会导致不一致，可用消息队列重试
- **先删缓存再更新数据库**：并发时可能读到旧数据回填缓存（延迟双删）
- **强一致性**：读写串行化、分布式锁、或用 Canal 监听 binlog 异步更新缓存

## Redis 应用场景总结？
- **缓存**：热点数据缓存，减轻数据库压力
- **分布式锁**：SETNX + 过期时间
- **计数器**：INCR 原子自增（文章阅读数、点赞数）
- **排行榜**：ZSet + ZREVRANGE
- **消息队列**：List（简单队列）、Stream（消费组，类似 Kafka）
- **限流**：ZSet 滑动窗口、令牌桶
- **Session 共享**：集群环境下统一会话存储
- **地理信息**：GEO 数据类型，附近的人/商铺

## Redis Pipeline 和事务？
- **Pipeline**：批量发送命令，减少网络 RTT，非原子性
- **MULTI/EXEC 事务**：命令入队后原子执行，不支持回滚（某条失败其他仍执行）
- **WATCH 乐观锁**：WATCH key 后，执行事务前检查 key 是否被修改，修改则事务失败
- **Lua 脚本**：原子执行多条命令，适合复杂逻辑，Redis 保证脚本执行期间不中断

## Redis 大 Key 问题？
- **定义**：单个 key 的 value 过大（如 >10KB 的 String，>5000 元素的集合）
- **危害**：阻塞其他命令（单线程）、网络传输慢、删除耗时（DEL 大 key 会卡顿）
- **解决**：拆分大 Key（分片存储）、使用 UNLINK 异步删除、设置合理过期时间
- **发现**：`redis-cli --bigkeys` 扫描、`MEMORY USAGE key` 查看内存占用
