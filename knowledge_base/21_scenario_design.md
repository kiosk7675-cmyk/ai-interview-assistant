# 后端场景题八股文

## 短链接系统设计
- **核心思路**：长 URL → 哈希/自增ID → 短码 → 302重定向
- **短码生成方案**：
  - 哈希算法（MurmurHash/CRC64）+ 进制转换（62进制：0-9a-zA-Z）
  - 数据库自增ID + 62进制编码
  - 预生成短码池（离线批量生成存Redis，用完再补）
- **存储选型**：MySQL（小规模）/ Redis + MySQL（大规模，Redis缓存热点映射）
- **重定向**：HTTP 302（可统计点击）优于 301（浏览器缓存，无法统计）
- **高并发优化**：布隆过滤器防重复、Redis缓存热点、读写分离

## 秒杀系统设计
- **核心挑战**：瞬时高并发、超卖、库存扣减一致性
- **架构方案**：
  - 前端：按钮防抖 + 验证码削峰 + CDN静态资源
  - 网关层：限流（令牌桶/漏桶）、IP黑白名单
  - 服务层：Redis预扣库存（Lua脚本保证原子性）→ MQ异步下单 → DB最终落盘
  - 数据层：乐观锁防超卖（`UPDATE stock SET count=count-1 WHERE id=? AND count>0`）
- **关键点**：Redis Lua脚本扣减库存 + RocketMQ事务消息保证最终一致性
- **防刷策略**：用户频次限制、设备指纹、风控规则引擎

## 消息推送系统设计
- **推送方式**：长连接（WebSocket）/ 短轮询 / 长轮询 / SSE
- **架构**：网关 → 推送服务 → 消息队列（Kafka）→ 推送分发 → 客户端
- **在线状态**：Redis Set/ZSet 维护用户在线列表，心跳检测
- **离线消息**：推送失败存DB，用户上线后拉取
- **批量推送**：分批+定时任务，避免瞬间打满带宽

## Redis 缓存雪崩/击穿/穿透
- **缓存雪崩**：大量Key同时过期 → 加随机过期时间 / 多级缓存 / 熔断降级
- **缓存击穿**：热点Key过期瞬间大量请求穿透 → 互斥锁 / 逻辑过期 / 永不过期+异步刷新
- **缓存穿透**：查询不存在的数据 → 布隆过滤器 / 缓存空值（短TTL）
- **应对流程**：先查布隆过滤器 → 缓存 → 互斥锁查DB → 回填缓存

## 接口性能优化（2s → 200ms）
- **排查思路**：监控/日志定位瓶颈 → 区分CPU密集/IO密集
- **常见优化**：
  - 数据库：慢SQL分析 → 加索引 → 查询优化 → 读写分离
  - 缓存：Redis缓存热点数据，注意缓存一致性（延迟双删/Canal监听binlog）
  - 异步化：非核心逻辑异步化（MQ、线程池）
  - 批量操作：N+1查询 → JOIN/IN批量查询
  - 并行化：多接口串行 → CompletableFuture 并行调用
  - 连接池：调整数据库/HTTP连接池参数

## 线上接口突然变慢排查
- **排查步骤**：
  1. 监控定位：哪个接口慢？从什么时候开始？
  2. 日志分析：慢SQL日志 / GC日志 / 错误日志
  3. 常见原因：
     - DB：慢查询（缺失索引/锁等待/数据量增长）
     - 缓存：缓存穿透/热点Key过期/Redis阻塞
     - 网络：带宽打满/DNS解析慢/连接池耗尽
     - JVM：Full GC频繁/OOM/线程死锁
  4. 解决方案：加索引 / 加缓存 / 扩容 / 优化代码逻辑

## MySQL 分库分表方案
- **何时分**：单表超2000万行 / 单库QPS超5000
- **分片策略**：
  - 范围分片：按时间/ID范围，易扩展但有热点
  - 哈希分片：按用户ID取模，数据均匀但扩容难
- **中间件**：ShardingSphere（客户端）/ MyCat（代理层）
- **跨分片问题**：
  - 跨分片JOIN → 冗余字段 / 应用层组装
  - 分布式ID → Snowflake / 号段模式
  - 分布式事务 → Seata / 最终一致性
- **扩容方案**：一致性哈希 / 停机迁移 / 双写迁移

## 单体→微服务拆分
- **拆分原则**：先按业务领域拆分（DDD限界上下文），避免按技术层拆
- **拆分步骤**：
  1. 先抽公共服务（用户/权限/配置）
  2. 按业务域拆核心服务（订单/商品/支付）
  3. 渐进式拆分：绞杀者模式，新功能走微服务，旧功能逐步迁移
- **关键问题**：
  - 服务间通信：同步（HTTP/gRPC）/ 异步（MQ）
  - 分布式事务：TCC / Saga / 最终一致性
  - 服务发现：Nacos / Consul
  - 配置管理：Nacos / Apollo
  - 链路追踪：SkyWalking / Jaeger

## 分布式锁实现方案
- **Redis实现**：SET NX EX（原子操作）→ 释放时Lua脚本验证持有者 → Redlock（多节点）
- **ZooKeeper实现**：临时顺序节点 → Watch机制 → 天然可重入、防死锁
- **对比**：
  - Redis：性能高，但主从切换可能丢锁（需Redlock）
  - ZK：强一致，但性能较低
  - etcd：Raft协议，兼顾性能和一致性
- **注意**：锁续期（看门狗）、可重入、防误删（UUID校验）

## 技术选型：MySQL vs MongoDB
- **MySQL**：关系型、事务ACID、结构化数据、复杂JOIN、数据一致性要求高
- **MongoDB**：文档型、灵活Schema、水平扩展容易、适合非结构化/半结构化数据
- **选型依据**：
  - 需要事务 + 复杂查询 → MySQL
  - 数据结构多变 + 高写入 + 无强事务 → MongoDB
  - 混合方案：核心业务MySQL + 日志/评论MongoDB

## 技术选型：Kafka vs RabbitMQ
- **Kafka**：高吞吐、分布式、持久化、大数据场景、消费者拉取模式
- **RabbitMQ**：低延迟、丰富路由、AMQP协议、中小规模、消费者推送模式
- **选型依据**：
  - 日志/流处理/大数据 → Kafka
  - 业务消息/延迟队列/复杂路由 → RabbitMQ
  - 吞吐量：Kafka 百万级 vs RabbitMQ 万级

## 内存持续增长排查（OOM）
- **排查步骤**：
  1. `jstat -gcutil` 观察 GC 频率和各区域使用率
  2. `jmap -histo:live` 查看活跃对象分布
  3. `jmap -dump:format=b,file=heap.hprof` 导出堆转储
  4. MAT/VisualVM 分析堆转储，找大对象和 GC Root 引用链
- **常见原因**：
  - 集合类只增不删（List/Map无上限增长）
  - ThreadLocal 泄漏（线程池中未remove）
  - 缓存无淘汰策略
  - 资源未关闭（连接/流）
- **解决**：加容量限制 / WeakHashMap / LRU淘汰 / 及时close

## 设计一个限流系统
- **算法**：
  - 固定窗口：简单但有临界突刺
  - 滑动窗口：平滑但内存开销大
  - 令牌桶：允许突发，匀速产生令牌
  - 漏桶：匀速处理，平滑流量
- **分布式限流**：Redis + Lua脚本 / Sentinel / 网关层限流
- **多级限流**：网关→服务→接口，逐层细化
- **降级策略**：限流后返回友好提示 / 排队等待 / 降级到缓存数据
