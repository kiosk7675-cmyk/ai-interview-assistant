# 计算机网络八股文

## OSI 七层模型和 TCP/IP 四层模型？
| OSI 七层 | TCP/IP 四层 | 协议示例 |
|---------|-----------|---------|
| 应用层 | 应用层 | HTTP, HTTPS, DNS, FTP, SMTP |
| 表示层 | 应用层 | SSL/TLS, JPEG, ASCII |
| 会话层 | 应用层 | Session, RPC |
| 传输层 | 传输层 | TCP, UDP |
| 网络层 | 网络层 | IP, ICMP, ARP, OSPF |
| 数据链路层 | 网络接口层 | Ethernet, PPP, MAC |
| 物理层 | 网络接口层 | 光纤, 双绞线 |

## TCP 和 UDP 的区别？
| 特性 | TCP | UDP |
|-----|-----|-----|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠传输（确认+重传） | 不可靠 |
| 传输方式 | 字节流 | 数据报 |
| 拥塞控制 | 有（慢启动、拥塞避免等） | 无 |
| 头部大小 | 20-60 字节 | 8 字节 |
| 适用场景 | 文件传输、HTTP | 视频直播、DNS、游戏 |

## TCP 三次握手过程和原因？
1. **C → S: SYN**：客户端发送 SYN=1, seq=x，进入 SYN_SENT
2. **S → C: SYN+ACK**：服务端回复 SYN=1, ACK=1, seq=y, ack=x+1，进入 SYN_RCVD
3. **C → S: ACK**：客户端发送 ACK=1, seq=x+1, ack=y+1，进入 ESTABLISHED
- **为什么三次**：确认双方收发能力正常、防止历史 SYN 连接浪费服务端资源
- **SYN Flood 攻击**：发送大量 SYN 不完成握手，耗尽半连接队列。防御：SYN Cookie

## TCP 四次挥手和 TIME_WAIT？
1. **主动方 → FIN**：请求关闭
2. **被动方 → ACK**：确认收到（被动方可能还有数据要发）
3. **被动方 → FIN**：被动方也关闭
4. **主动方 → ACK**：确认后进入 TIME_WAIT（2MSL）
- **TIME_WAIT 原因**：确保最后一个 ACK 到达；等待网络残留报文消失
- **大量 TIME_WAIT**：高频短连接导致。解决：连接池复用、`tcp_tw_reuse`、增加端口范围

## TCP 拥塞控制？
- **慢启动**：cwnd 从 1 开始指数增长（每 RTT 翻倍）
- **拥塞避免**：cwnd 达到 ssthresh 后线性增长（每 RTT +1）
- **快重传**：收到 3 个重复 ACK 立即重传（不等超时）
- **快恢复**：快重传后 cwnd 减半（而非从1开始），ssthresh = cwnd/2

## TCP 滑动窗口？
- **发送窗口**：已发送但未确认的数据范围，接收方通过 ACK 通告窗口大小
- **接收窗口**：接收方缓冲区剩余空间，通告给发送方控制发送速率
- **流量控制**：接收方通过通告窗口大小限制发送方速率，防止接收方缓冲区溢出
- **零窗口探测**：接收方通告窗口为0时，发送方定期发探测报文，避免死锁

## HTTP 状态码？
- **1xx**：信息性（100 Continue）
- **2xx**：成功（200 OK, 201 Created, 204 No Content）
- **3xx**：重定向（301 永久重定向, 302 临时重定向, 304 未修改缓存）
- **4xx**：客户端错误（400 Bad Request, 401 未认证, 403 禁止, 404 未找到）
- **5xx**：服务端错误（500 内部错误, 502 网关错误, 503 服务不可用, 504 网关超时）

## HTTP 和 HTTPS 区别？
- **HTTP**：明文传输，端口 80，不安全
- **HTTPS**：TLS 加密传输，端口 443，需要 CA 证书
- **TLS 握手**：客户端发送支持的加密套件 → 服务端选一个+发证书 → 客户端验证证书 → 协商对称密钥 → 加密通信
- **性能开销**：TLS 握手增加 1-2 个 RTT（TLS 1.3 可 0-RTT），加解密 CPU 开销

## Cookie、Session、Token 区别？
- **Cookie**：浏览器自动携带，4KB 限制，有 CSRF 风险。HttpOnly 防 XSS，SameSite 防 CSRF
- **Session**：服务端存储会话数据，Session ID 通过 Cookie 传递，服务端需存储（集群需 Redis 共享）
- **Token（JWT）**：无状态，服务端不存储，客户端携带 Token 请求。适合分布式/微服务
- **JWT 结构**：Header.Payload.Signature，Payload 存用户信息，签名防篡改

## DNS 解析过程？
1. 浏览器缓存 → 2. OS 缓存 → 3. 路由器缓存 → 4. ISP DNS → 5. 根域名服务器 → 6. 顶级域名服务器 → 7. 权威域名服务器
- **递归查询**：客户端向本地 DNS 服务器查询
- **迭代查询**：本地 DNS 服务器依次向根/顶级/权威服务器查询
- **DNS 劫持**：篡改 DNS 响应指向恶意 IP。防御：DNSSEC、HTTPS

## 从输入 URL 到页面显示的过程？
1. DNS 解析获取 IP
2. TCP 三次握手建立连接
3. TLS 握手（HTTPS）
4. 发送 HTTP 请求
5. 服务器处理请求返回响应
6. 浏览器解析 HTML → 构建 DOM 树
7. 解析 CSS → 构建 CSSOM 树
8. DOM + CSSOM → 渲染树 → 布局 → 绘制
9. 执行 JS（可能阻塞渲染）
10. TCP 四次挥手关闭连接

## ARP 协议？
- **作用**：将 IP 地址解析为 MAC 地址
- **过程**：主机广播 ARP 请求（谁的 IP 是 X？），目标主机单播 ARP 响应（我是 X，MAC 是 Y）
- **ARP 缓存**：存储 IP-MAC 映射，有超时时间
- **ARP 欺骗**：伪造 ARP 响应，将流量导向攻击者。防御：静态 ARP 绑定、DAI
