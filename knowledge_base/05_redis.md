# Redis 缓存

## Redis 为什么快？
1. **纯内存操作**：数据存储在内存，读写延迟微秒级。
2. **单线程模型**：避免线程切换和锁竞争开销，命令顺序执行。
3. **I/O 多路复用**：epoll 机制，单线程处理大量并发连接。
4. **高效数据结构**：SDS（预分配+惰性释放）、跳表（O(logN)）、压缩列表（小数据节省内存）、quicklist。

## Redis 五种基础数据结构及使用场景？
- **String**：缓存、计数器（INCR）、分布式锁（SETNX）、Session。
- **Hash**：对象存储（用户信息、商品详情），`HGET/HSET` 操作单个字段。
- **List**：消息队列（LPUSH + BRPOP）、最新列表（时间线）。
- **Set**：去重、交集/并集（共同好友、标签系统）、抽奖（SRANDMEMBER）。
- **Sorted Set（ZSet）**：排行榜（分数排序）、延迟队列（时间戳作分数）、滑动窗口限流。

## Redis 持久化？
- **RDB**：定时快照，fork 子进程写磁盘。恢复快，但可能丢失两次快照间的数据。
- **AOF**：追加写日志，三种同步策略：always（每条）、everysec（每秒，推荐）、no（OS 决定）。
- **AOF 重写**：合并冗余命令，缩小文件体积。BGREWRITEAOF 异步执行。
- **混合持久化（4.0+）**：AOF 重写时前半段用 RDB 格式，后半段用 AOF，兼顾恢复速度和数据安全。
- **生产建议**：RDB + AOF 混合，RDB 做冷备，AOF 做实时保护。

## Redis 缓存三大问题？
- **缓存穿透**：查询不存在的数据，请求直达数据库。
  - 解决：布隆过滤器（拦截不存在的 key）、缓存空值（设短 TTL）。
- **缓存击穿**：热点 key 过期瞬间，大量请求打到数据库。
  - 解决：互斥锁（SETNX 抢锁重建）、热点 key 永不过期（逻辑过期）。
- **缓存雪崩**：大量 key 同时过期，或 Redis 宕机。
  - 解决：过期时间加随机偏移、Redis 集群高可用、多级缓存、限流降级。

## Redis 过期删除策略？
- **惰性删除**：访问 key 时检查是否过期，过期则删除。CPU 友好但内存不友好。
- **定期删除**：每 100ms 随机抽取部分 key 检查过期，过期则删除。折中方案。
- **实际策略**：两者结合。如果只靠惰性删除，大量过期 key 不被访问会占用内存。

## Redis 内存淘汰策略？
- **noeviction**：默认，内存满拒绝写入。
- **allkeys-lru**：所有 key 中淘汰最久未使用的（推荐）。
- **volatile-lru**：设置了过期时间的 key 中淘汰最久未使用的。
- **allkeys-random / volatile-random**：随机淘汰。
- **volatile-ttl**：淘汰 TTL 最短的。
- **allkeys-lfu（4.0+）**：淘汰使用频率最低的（推荐替代 LRU）。

## Redis 集群方案？
- **主从复制**：一主多从，读分离。主写从读，主宕机需手动切换。
- **哨兵（Sentinel）**：监控主从，自动故障转移。至少 3 个哨兵节点。
- **Cluster**：官方分片方案，16384 个槽位分布到多个节点，支持水平扩展。每个主节点有从节点。
- **Cluster 限制**：不支持多 key 事务（跨槽）、mget 需 hash tag 保证同槽、运维复杂。

## Redis 分布式锁？
- **基本实现**：`SET key value NX EX 30`（加锁+过期原子操作）。
- **value 使用 UUID**：防止误删其他客户端的锁，释放时先比较再删除（Lua 脚本保证原子性）。
- **Redlock**：多节点分布式锁，在 N/2+1 个节点获取锁才算成功。争议较大，多数场景单节点锁足够。
- **Redisson**：Java 客户端，看门狗自动续期，可重入锁，生产级分布式锁方案。
- **Python 实现要点**：`redis-py` 的 `set(nx=True, ex=30)`，释放用 Lua：`if redis.call('get',KEYS[1])==ARGV[1] then return redis.call('del',KEYS[1]) else return 0 end`。

## Redis 与数据库一致性？
- **Cache Aside 模式**：读缓存未命中则查数据库并写缓存；写操作先更新数据库再删缓存。
- **延迟双删**：写数据库 → 删缓存 → 延迟 N 毫秒 → 再删缓存。防止旧数据被并发读回缓存。
- **订阅 binlog**：通过 Canal 等中间件监听 MySQL binlog，异步更新缓存，最终一致性。
- **强一致性**：Redis 无法保证强一致，需要分布式事务或直接查数据库。

## Redis 实际应用场景？
- **登录防暴力破解**：`INCR login_fail:{ip}` + `EXPIRE` 60 秒，超限则拒绝。
- **接口限流**：滑动窗口（ZSet 时间戳作分数）、令牌桶。
- **排行榜**：ZSet + ZREVRANGE。
- **延迟任务**：ZSet 用执行时间作分数，定时器 ZRANGEBYSCORE 取出到期的任务。
- **分布式 Session**：Spring Session / 自定义中间件存 Redis。
